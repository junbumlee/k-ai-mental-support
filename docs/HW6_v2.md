## 2. v1에 대한 비판 — 채워야 할 빈칸

### 2.1 "리더 카테고리"가 표면적일 위험
v1은 카테고리 dropdown 추가에서 멈출 수 있다. 그러나 MANIFESTO의 약속은 단순 분류가 아니라 **"리더 특유의 인지 오류 포착"** + **"상황별 AI 가이드"**다. dropdown만으로는 일반 감정 앱에 라벨 하나 붙인 것과 구분이 안 된다.

→ **개선**: 카테고리는 1) UI 선택, 2) 시스템 프롬프트의 카테고리별 분기, 3) 피드백 카드의 카테고리 컨텍스트 표기 — **3단 통합**이어야 한 개의 의미 있는 feature가 된다.

### 2.2 CBT 원칙과의 잠재적 충돌이 빠져있다
[CLAUDE.md](../CLAUDE.md)의 안전장치 #2: *"AI는 사고를 대신 재구성하지 않고 질문으로 돌려준다 / 진단 금지 / 판단·훈계 금지"*.

한편 [Premortem.md](./Premortem.md) 시나리오 2의 대응책: *"내일 바로 쓸 수 있는 대화 스크립트 등 구체적 솔루션 제공"*.

→ 이 둘은 **일부 충돌한다**. "솔루션 제시"가 "사고 대신 재구성"이 되면 CBT 원칙이 깨진다.
→ **개선**: 카테고리별 프롬프트 변형은 **"왜곡 유형 후보를 좁히고 되묻는 질문의 결을 바꾸는 것"**까지만. *"내일 이렇게 하세요"* 액션 처방은 이번 feature 범위에서 의도적으로 제외 (별도 PR/단계로 분리).

### 2.3 측정 지표가 없다
"리더 전용처럼 보이게 됐다"는 검증 불가능한 자기 평가다.

→ **개선**: 팀원 리뷰 시 던질 **고정 질문 3개** 사전 준비 (섹션 5 참조).

### 2.4 실패/롤백 시나리오 부재
- /office-hours가 카테고리와 무관한 다른 액션을 제안하면?
- 카테고리별 시스템 프롬프트 변형이 LLM 응답 품질을 오히려 떨어뜨리면?
- 기존 localStorage 기록(카테고리 없음)과 호환은?

→ **개선**: 섹션 6에 명시.

### 2.5 타임박스가 없다
v1의 "Step 1~5"는 무한히 늘어질 수 있다.

→ **개선**: 각 단계 ≤ N분으로 박스 (섹션 7).

### 2.6 카테고리 후보 자체에 MANIFESTO와의 정렬 미흡
v1 카테고리: 성과압박 / 팀원관리 / 평가면담 / 상사보고 / 팀내갈등.
MANIFESTO 명시 카테고리: **성과압박 / 팀원관리(고과·근태) / 팀장 간 경쟁**.

→ **개선**: MANIFESTO 표기와 정확히 맞추되 v1의 추가 항목(평가면담·상사보고·팀내갈등) 중 1~2개만 흡수해 5개 이내로. **"기타/직접 입력"** 옵션도 두어 분류 강제로 인한 입력 마찰 방지.

## 3. 통합 개선 계획

### 3.1 핵심 이야기 (제출 시 일관되게 유지할 한 줄)

> *"우리는 리더용 도구를 표방했지만 MVP는 일반 감정 앱처럼 보였다. /office-hours로 이 간극을 짚었고, 제품 정체성을 가장 빠르게 선명하게 만드는 작은 기능 — 리더 상황 카테고리 + 카테고리 인지 프롬프트 — 을 골라 별도 브랜치에 구현했다."*

### 3.2 Feature 정의 (확정)

**이름**: `리더 상황 카테고리 + 카테고리-인지 프롬프트 변형`

**구성 (3단 통합)**:
1. **입력 UI**: 카테고리 5개 + 기타. 단일 선택. (성과 압박 / 팀원 관리 / 평가·고과 / 상사·보고 / 팀 내 갈등)
2. **데이터·API**: 클라이언트 payload + localStorage 스키마에 `category` 필드 추가. 기존 기록은 `null` 허용 (무마이그레이션).
3. **프롬프트 변형**: `api/index.py`의 `SYSTEM_PROMPT`는 그대로 두고, **유저 메시지 블록 앞에 카테고리 컨텍스트 한 단락만 prepend**. 시스템 프롬프트의 3원칙은 건드리지 않는다 (CBT 안전장치 보호).

**범위 밖 (의도적 제외)**:
- "내일 바로 써볼 행동 스크립트" 생성 → CBT 원칙 충돌, 별도 단계
- 카테고리별 통계/대시보드 → 다음 feature
- 카테고리 다중 선택 → 진단 정확도보다 입력 마찰만 키움

### 3.3 구현 영향 영역 (파일 매핑)

| 파일 | 변경 내용 |
|---|---|
| `templates/index.html` | "쓰기" 탭 입력 위에 카테고리 칩(또는 select) 추가 |
| `static/app.js` | 폼 상태에 `category` 추가, localStorage 저장 객체 확장, 기록 카드에 카테고리 뱃지 렌더 |
| `api/index.py` | `AnalyzeRequest`에 `category: Optional[str]` 추가, `_build_user_block()`(없으면 신설)에서 카테고리별 prefix 1~2문장 삽입. `SYSTEM_PROMPT` **변경 금지** |
| `vercel.json` | 변경 없음 |

## 4. /office-hours 운영 계획

### 4.1 입력할 컨텍스트 (v1 초안 + 보강)

```md
타겟: 신임 팀장 / 팀장 후보자
문제: 리더십 불안, 샌드위치 스트레스, 고립감
현재 MVP: 3줄 기록(상황/자동화 사고/재구성) + LLM CBT 피드백 + 음성 입력 + 위기 키워드 필터 + localStorage
간극: MANIFESTO는 "리더 특유 인지 오류 포착"을 약속하지만 현 MVP는 일반 감정 앱과 구분이 약함.
CBT 안전장치: AI는 사고를 대신 재구성하지 않고 되묻기만 한다 (이 원칙은 양보 불가).

검토 요청:
1. 타겟을 더 좁힐 여지 (팀장 vs 팀장 후보자 vs 신임 1년차 등)
2. "리더 도구임이 첫 화면에서 인식되게" 만드는 최소 변경 1개
3. CBT 원칙(되묻기만)과 Premortem이 요구하는 "구체적 솔루션" 사이의 절충 지점
4. 향후 12개월 내 가장 먼저 깨질 가설은 무엇인가
```

### 4.2 백업 시나리오
office-hours가 카테고리와 **무관한** 액션을 강하게 권하면:
- 그 액션이 1주 안에 코드로 실현 가능한가? → Yes면 그쪽으로 선회
- No면 → "흥미로운 제안이지만 본 과제 범위 밖"으로 분류하고 카테고리 feature 진행. 제출 답변에 *"이 제안은 다음 단계로 보류"* 라고 솔직히 명시 (오히려 메타인지 점수 ↑)

## 5. 팀원 리뷰 — 고정 질문 3개

브랜치 공유 시 동일하게 던진다 (응답 비교 가능):

1. **정체성**: "이 화면을 처음 본 사람이 '리더용 도구'라고 1초 안에 알아챌 것 같은가? Yes/No + 이유 한 줄."
2. **마찰**: "카테고리 선택이 '도움이 되는 가이드'로 느껴지는가, '귀찮은 분류 강요'로 느껴지는가?"
3. **CBT 원칙**: "피드백이 여전히 '되묻기' 결을 유지하고 있는가, 아니면 훈계조로 바뀌었는가?"

→ 3번은 직접 정성 평가 + 같은 입력으로 main 브랜치 vs feature 브랜치 응답 비교 스크린샷.

## 8. 제출 답변 — 일관된 스토리로 묶기

v1 섹션 8을 거의 그대로 쓰되, **모든 답변에 "정체성 간극 → 카테고리 feature" 한 줄을 공통 베이스로 깔기**.

### 8.1 Biggest lesson
> "기능을 추가하기 전에 타겟이 좁혀져야 한다. /office-hours는 코드 리뷰가 아니라 **타겟 좁히기 도구**였다."

### 8.2 Action items (1개 선택)
- (a) 타겟 정의 좁히기 — 문서 작업
- (b) **리더 상황 카테고리 추가 — 선택**
- (c) 프라이버시 메시지 전면화 — 카피 작업
- (d) "내일 행동" 강화 — CBT 원칙 충돌, 보류

→ (b) 선택. 이유: 코드와 제품 정체성을 동시에 움직이는 단 하나의 작업.

### 8.3 Result
- main과 feature 브랜치 응답 비교 스크린샷 1쌍
- "리더 상황을 명시하니 LLM이 왜곡 유형을 더 좁게 짚었다" 같은 관찰 1줄 (실제 결과 보고)

### 8.4 Branch URL
PR 생성 후 GitHub URL.

### 8.5 Biggest challenge
> "UI 추가는 30분이지만, 그 데이터가 LLM 응답에 의미 있게 반영되게 하는 데 더 오래 걸렸다. 게다가 시스템 프롬프트는 CBT 원칙 때문에 못 건드려서, 변경 지점을 user 블록 prefix로 한정해야 했다."

### 8.6 Meta-cognition lesson
> "AI에게 '뭘 만들까'를 물으면 일반론을 받는다. '이 두 원칙 사이에서 어디까지 양보할까'를 물으면 우리 제품에 맞는 답을 받는다."

### 8.7 Technical lesson
> "프롬프트 엔지니어링은 시스템 프롬프트를 건드리는 것만이 아니다. user 블록 구조를 바꾸는 것이 더 안전하면서도 효과가 컸다."

### 8.8 Team responses
섹션 5의 3개 질문 답변 원문 + 한 줄 종합.
