# HW6 접근 방식과 실행 계획

## 1. 과제 핵심 이해

이번 과제는 두 가지를 요구한다.

1. `gstack`의 `/office-hours`를 사용해 현재 프로젝트의 방향을 AI와 함께 검토한다.
2. 그 검토 결과를 바탕으로, 별도 브랜치에서 기능 하나를 직접 만들고 AI 및 팀원 리뷰까지 연결한다.

즉, 과제의 본질은 단순 구현이 아니라 아래 흐름을 증명하는 것이다.

`아이디어 검토 -> 방향 수정 -> 기능 선택 -> 별도 브랜치 구현 -> 리뷰 -> 회고`

따라서 이번 과제에서 가장 중요한 질문은 이것이다.

## 3. 과제 접근 방식

### 접근 원칙

이번 과제는 아래 원칙으로 접근한다.

1. `/office-hours`는 아이디어를 칭찬받는 자리가 아니라 방향을 더 좁히는 자리로 사용한다.
2. 기능 구현은 큰 기능보다 `방향성을 더 선명하게 만드는 작은 기능`을 선택한다.
3. 제출 답변은 "AI가 이런 말을 했다"보다 "그래서 무엇을 바꿨다"에 초점을 둔다.

### 왜 이 접근이 맞는가

- 현재 프로젝트는 비전 자체는 이미 있다.
- 부족한 것은 `누구를 위한 제품인지`와 `왜 이 제품이어야 하는지`를 MVP에서 체감하게 만드는 부분이다.
- 따라서 `/office-hours`에서 제품 방향을 압축하고, 기능 구현도 그 압축된 방향을 반영하는 것이 가장 설득력 있다.


## 4. `/office-hours`에서 확인할 핵심 질문

`/office-hours`에 가져갈 핵심 맥락은 아래와 같이 정리한다.

### 입력할 프로젝트 설명 초안

```md
우리는 신임 팀장과 팀장 후보자의 리더십 불안과 심리적 소진을 돕는 AI 서비스를 만들고 있다.
현재 MVP는 세 줄 기록, 음성 입력, AI 기반 CBT 피드백 기능이 있다.
하지만 지금 상태는 일반적인 감정 기록 앱처럼 보일 수 있다.

우리가 검토받고 싶은 것은:
1. 타겟을 얼마나 좁혀야 하는가
2. 가장 설득력 있는 초기 사용 시나리오는 무엇인가
3. 현재 MVP에 추가하면 '리더 전용 서비스'처럼 보이게 만드는 최소 기능은 무엇인가
```

### 기대하는 질문 방향

`/office-hours`에서 특히 확인해야 할 것은 다음이다.

- 지금 사용자가 정말 `직장인 전체`인지 `신임 리더`인지
- 사용자가 현재 이 문제를 어떤 방식으로 버티고 있는지
- 가장 절박한 순간이 무엇인지
- 지금 앱이 해결하는 문제 단위가 너무 넓지 않은지
- 미래에도 이 제품이 더 중요해질 이유가 있는지


## 5. `/office-hours` 이후 예상 액션 아이템

AI 리뷰 결과로 나올 가능성이 높은 액션 아이템은 아래와 같다.

1. 타겟을 `직장인 전체`에서 `신임 팀장/팀 리더`로 더 좁힌다.
2. 입력 경험에 `리더 상황 맥락`을 추가한다.
3. 피드백 결과가 단순 위로가 아니라 `내일 바로 써볼 행동` 중심이 되도록 강화한다.
4. 프라이버시와 데이터 신뢰 메시지를 더 전면에 드러낸다.

이 중 이번 과제에서 실제 구현 대상으로 가장 적절한 것은 `리더 상황 맥락 추가`다.


## 6. 구현할 기능 제안

### 선택 기능

`리더 고민 카테고리 선택 기능`

예시 카테고리:

- 성과 압박
- 팀원 관리
- 평가 면담
- 상사 보고
- 팀 내 갈등

### 이 기능을 선택하는 이유

1. 현재 제품의 정체성을 가장 빠르게 선명하게 만든다.
2. 구현 범위가 과제용 브랜치 작업으로 적절하다.
3. 프론트엔드와 프롬프트 모두에 자연스럽게 연결된다.
4. 제출 문항에서 "방향 검토 결과 실제로 반영한 기능"이라고 설명하기 쉽다.

### 예상 구현 방식

- 입력 화면에 카테고리 선택 UI 추가
- 선택한 카테고리를 저장 데이터에 포함
- AI 분석 요청 payload에 카테고리 전달
- 시스템 프롬프트 또는 사용자 블록에 카테고리 맥락 반영
- 기록 화면에서도 해당 카테고리 표시

이번 과제의 목표는 단순히 "기능 하나 만들었다"가 아니다.

이 스토리가 분명하면, 과제 완성도가 높아진다.
